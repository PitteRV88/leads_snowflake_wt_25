Fecha: 2026-03-30
Commit: 108eadc

### Cambios realizados en sesion 9:

#### 1. Dialog reopen pattern (refresco de datos sin cerrar):
- Problema: En sesion 8 se elimino st.rerun() para que el dialog no se cerrara,
  pero esto causaba que los datos mostrados quedaran desactualizados (stale).
  El usuario no podia encadenar acciones (ej: guardar URL -> generar pitch)
  sin cerrar y reabrir manualmente el dialog.
- Solucion: Patron "reopen" - en cada accion exitosa:
  1. st.cache_data.clear() -> invalida cache de datos
  2. st.session_state["_open_cuenta"] = acct_name -> marca para reabrir
  3. st.session_state["_toast_msg"] = "mensaje" -> guarda notificacion
  4. st.rerun() -> reinicia el script
  El mecanismo existente en linea ~1387 detecta _open_cuenta y reabre el dialog
  automaticamente con datos frescos. El cierre+reapertura es instantaneo.
- Aplicado a las 7 acciones del dialog:
  * Marcar como Contactado
  * Agregar Contacto
  * Cambiar Contacto Principal
  * Editar Datos Cuenta
  * Registrar Interaccion
  * Guardar accion y marcar contactados (Pitch IA)
  * Descalificar Lead

#### 2. Notificaciones toast verdes:
- Agregado st.toast() (Streamlit >= 1.55) despues de load_data()
- Consume _toast_msg del session_state y muestra notificacion flotante con icono verde
- Se desvanece automaticamente en ~5 segundos
- Reemplaza los anteriores st.success() estaticos dentro del dialog

#### 3. Fix de paginacion con filtros KPI:
- Bug: Al hacer click en un KPI (ej: "Contactados") y luego en "Siguiente",
  la paginacion no cambiaba - siempre mostraba la misma pagina.
- Causa raiz: st.session_state["leads_page"] = 0 se ejecutaba en cada rerun
  cuando _kpi_filter existia en session_state. Como el filtro nunca se eliminaba
  (solo con "Limpiar filtro"), cada rerun reseteaba la pagina a 0.
- Solucion: Comparar _kpi_filter con _kpi_filter_prev. Solo resetear leads_page
  cuando el filtro cambia (nuevo filtro o filtro diferente), no en cada rerun.
  Al limpiar filtro, se limpia tambien _kpi_filter_prev.
